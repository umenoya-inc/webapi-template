## 概要

テストが通ることを確認した上で、テストコードの `describe`/`it` ブロックから関数の振る舞い仕様を読み取り、ソースファイル JSDoc の概要行と振る舞い（本文）を生成・更新する。

JSDoc の全体構成は [JSDoc フォーマット](docs/rule/jsdoc-format.md) ルールに従う。
このスキルが責任を持つのは **概要行（先頭行）と振る舞い（本文の箇条書き）** のみであり、`@remarks` など他のセクションには一切触れない。

## 手順

### 1. 対象ファイルの特定

`$ARGUMENTS` にソースファイルのパス（例: `src/modules/db/user/createUser.ts`）が渡される。

対応するテストファイルは同ディレクトリの `<basename>.test.ts` とする。
テストファイルが見つからない場合はユーザーに報告して終了する。

### 2. テストの実行

```bash
npx vp test $TEST_FILE
```

テストが失敗した場合はコメント生成を行わず、失敗内容をユーザーに報告して終了する。

### 3. テストコードの読み取り

テストファイルを読み取り、`describe`/`it` ブロックの構造とテスト名を把握する。
各テストケースの中身（アサーション）も確認し、具体的な振る舞いを正確に理解する。

### 4. 振る舞い（本文）の生成

テストから読み取った情報を元に、概要行直後の箇条書きを生成する。

ルール:

- 型情報（引数の型、戻り値の型、スキーマ制約）は書かない
- テストの `it` 記述とアサーション内容を元に、抽象度の高い仕様表現にする
- `it` の日本語記述をそのままコピーするのではなく、仕様として自然な表現に整える
- 成功ケースを先に、エラーケースを後に並べる

### 5. ソースファイルの更新

既存の JSDoc がある場合:

- **概要行（先頭行）** を更新する
- **振る舞いの箇条書き** が既にあれば差し替え、なければ概要行の直後に挿入する
- **`@remarks` やその他の手書きセクション** はそのまま保持する

JSDoc が存在しない場合:

- [JSDoc フォーマット](docs/rule/jsdoc-format.md) に従い、概要行と振る舞いの箇条書きだけを持つ JSDoc を export 文の直前に挿入する
