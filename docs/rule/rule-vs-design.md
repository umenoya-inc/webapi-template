# rule と design の区分基準

## rule（docs/rule/）

違反か準拠かを判定できる具体的な制約。「こう書け」「これはするな」。

- ファイル構成、命名規則、コーディングパターン、API の使い方など
- 理想的には lint で機械的に強制可能
- 新しいコードを書くとき・レビューするときのチェックリストとして機能する

## design（docs/design/）

設計判断の根拠となるアーキテクチャ方針。「なぜこの構造か」。

- モジュールの責務分離、モデリングの方針、テスト戦略など
- 新しい機能をどの層に置くか、どう設計するかの指針として機能する
- 個別の書き方ではなく、構造全体の意思決定を扱う

## 判断基準

- コードの書き方に対して「正しい/正しくない」が言えるなら **rule**
- 設計上の選択肢に対して「なぜこちらを選んだか」を説明しているなら **design**
- 両方の性質を持つ場合は分離する。具体的な制約を rule に、動機や方針を design に置く
