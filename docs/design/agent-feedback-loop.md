# コーディングエージェントのための高速フィードバック設計

コーディングエージェントが品質を保ちながら高速にコードを書くために、フィードバックサイクルを最適化する。このプロジェクトの設計方針・ルール・検証手法は、すべてこの上位方針の下に位置付けられる。

## 原則

- **フィードバックは速く返す**: エージェントが間違った方向に進んだら、できるだけ早い段階で検出する
- **フィードバックは曖昧にしない**: 何が問題で、どう直すべきかが明確に伝わる
- **人間が判断するステップを最小化する**: エージェントが `check → test → 修正` のループを自力で回せる状態を目指す

## フィードバックの階層

速度と自動化度合いに応じて5つの階層がある。上位ほどフィードバックが速く、曖昧さが少ない。

### 1. コンパイル時フィードバック

型制約によって、コードが書かれた瞬間にエラーを返す。最も速く、最も曖昧さがない。

- **型によるテスト網羅の強制**（[type-enforced-test-coverage](type-enforced-test-coverage.md)）: Desc ラベルとtestBehavior によりテストケースの過不足がコンパイルエラーになる
- **ブランド型によるドメイン制約**（[branded-entity-id](../rule/branded-entity-id.md)）: `UserId` と `string` を型レベルで区別し、取り違えを防ぐ
- **エラーハンドリング**（[error-handling](../rule/error-handling.md)）: 想定内の失敗は DU で型表現しハンドリング漏れをコンパイルエラーにする。想定外のバグは Let it crash で一律 500 + 構造化ログ

### 2. lint 時フィードバック（実装コード）

書いたコードの構造的な制約を自動検証する。`npx vp check` で即座に実行される。

- **1ファイル1エクスポート**（[one-export-per-file](../rule/one-export-per-file.md)）: ファイルの責務を単一に保ち、変更の影響範囲を明確にする
- **モジュール境界の強制**（[module-barrel-export](../rule/module-barrel-export.md)、[module-responsibility](module-responsibility.md)）: モジュール間の依存を barrel export に限定し、内部実装への依存を防ぐ
- **Namespace Import の禁止**（[no-namespace-import](../rule/no-namespace-import.md)）: 依存先を明示的にし、使用箇所の追跡を容易にする

### 3. lint 時フィードバック（型定義）

書いた型自体の品質を自動検証する。実装コードではなく型定義を検査対象とするメタレベルの検証。

- **型の展開コスト**（[type-expansion-cost](type-expansion-cost.md)）: 型の AST を走査し、展開コストが閾値を超えた型を検出する

この階層は始まったばかりだが、型の事後検証（[type-post-validation](../ideas/2026-03-22-type-post-validation.md)）や危険な型パターンの検出（[dangerous-type-patterns](../ideas/2026-03-22-dangerous-type-patterns.md)）など、拡充の方向性が見えている。

### 4. テスト時フィードバック

実行して振る舞いを検証する。コンパイル時・lint 時では検出できない実行時の問題を捕捉する。

- **testBehavior による振る舞い検証**: 型で網羅を強制された上で、各コードパスの実際の動作を検証する
- **DB 統合テスト**: 実際のデータベースに対してクエリを実行し、スキーマとの整合性を検証する

### 5. 対話的フィードバック

自動検証では判断できない定性的な問題を、人間またはスキルによる対話的な検査で発見する。

- **check-hover スキル**: LSP hover の可読性を定性的に検証し、lint では拾えない truncation・内部構造の露出・二重展開を検出する
- **人間によるレビュー**: 設計意図との整合性、命名の適切さ、ドメインモデルの妥当性など

## 検証手法の進化プロセス

検証手法は以下の流れで成熟する。指標を先に設計するのではなく、実際に問題に遭遇してパターンが見えてから定量化する。

1. **問題の発見**: 対話的な検査や開発中の体験を通じて、繰り返し発生する問題を認識する
2. **パターンの抽出**: 複数の事例から共通するパターンを特定し、アイディアメモとして記録する
3. **定量化**: パターンをスカラー値や判定基準として定式化する
4. **lint への昇格**: 定量化した指標を lint ルールとして実装し、自動検出に移行する

この流れ自体が check-hover → type-expansion-cost の実例として検証済み。

## エージェントにとっての意味

この設計により、コーディングエージェントは以下のループを自律的に回せる。

1. コードを書く
2. `npx vp check --fix` で構造・型の検証（階層 1〜3 をまとめて実行）
3. `npx vp test` で振る舞いの検証（階層 4）
4. エラーがあれば修正して 1 に戻る

人間の介入が必要なのは階層 5 のみ。階層 1〜4 で品質の大部分をカバーすることで、人間は設計判断やドメイン知識の提供に集中できる。
