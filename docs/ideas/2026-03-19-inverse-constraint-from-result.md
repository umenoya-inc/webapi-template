# Inverse Constraint: 結果から入力を逆方向に制約する設計パターン

## 概要

通常のプログラミングでは「入力 → 結果」の順で考えるが、この設計では「結果の宣言（Desc ラベル）」が起点となり、テストの入力側が逆方向に制約される。3DCG の Inverse Kinematics（手先の位置から関節の角度を逆算する）に似た構造。

## 現在の具体例

1. **`okAs` / `failAs` でパスを宣言**（結果の定義）
2. **`InputScenarios` でパスに到達する入力条件を列挙**（結果 → 入力の逆方向制約）
3. **`parameterize` / `propertyCheck` のキーがシナリオラベルで強制される**（制約の型レベル伝播）

```
結果の宣言 (Desc)
  → パスに到達する入力条件 (InputScenarios)
    → テストのパラメータキー (parameterize / propertyCheck)
      → 具体的な入力値 / arbitrary
```

## 応用候補

- **ドキュメント自動生成** — 各パスの到達条件が型に埋め込まれているため、API ドキュメントに「このエラーはどういう入力で発生するか」を自動的に記述できる可能性がある
- **テスト生成の自動化** — シナリオラベルから「何をテストすべきか」のスケルトンを自動生成する（具体値は人間が埋める）
- **契約プログラミングとの接続** — 事前条件（入力制約）を事後条件（結果のパス）から逆算する考え方は、Design by Contract の逆方向バリエーションとも見れる

## 背景

`parameterize` の実装から始まり、「実装側がテスト側のキーを制約できないか」という問いを経て、`InputScenarios` が生まれた。結果として、`Desc` ラベルという一つの概念を軸に、テスト網羅（testBehavior）、モック網羅（mockBehavior）、入力パターン網羅（parameterize / propertyCheck）がすべて同じ方向に統一された。この「結果から入力を制約する」パターンは意図して設計したものではなく、段階的に発展した結果として自然に現れた。
