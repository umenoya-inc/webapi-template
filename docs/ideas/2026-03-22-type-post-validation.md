# 型の事後検証アプローチの応用

## 概要

型定義自体を検査対象として扱い、書いた型の健全性を事後的に検証する。型でコードを制約する（型→コード）のではなく、型自体の品質を検査する（型→型）というメタレベルの検証。

## 応用候補

- **union の構造的一貫性**: discriminated union の各分岐が一貫した構造を持っているか、分岐数が過剰でないか
- **ジェネリクスの伝播距離**: 型パラメータが何段の関数呼び出しを経て解決されるか。距離が長いほど型エラーメッセージが不可解になる
- **ブランド型の一貫性**: 同じブランド名が異なるベース型で使われていないか

## 背景

- `type-expansion-cost` lint ルールの実装を通じて、型の AST 走査による定量的な検証が実用的であることを確認した
- 重要な非対称性: 「型を安全に書くための事前制約」ではなく「書いた型が健全かの事後検証」であること。循環的複雑度と同じ立ち位置
- 発見のプロセス: 目視で問題を発見（check-hover）→ パターンが見えたら定量化して lint に昇格。次の応用候補も同じ流れで、実際に困ったケースが溜まってから定量化する
