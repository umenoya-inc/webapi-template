# 危険な型のパターン

## 概要

型定義が技術的には正しく動いていても、保守性・デバッガビリティ・パフォーマンスの面で問題を引き起こす型のパターンをまとめる。実用性は LLM が担保できる時代において、それ以外の側面で型設計の品質を考えるための指針。

事後検証の原則に基づき、実際に問題に遭遇したケースが溜まった時点で定量化・lint 化を検討する。

## 1. 型エラーのデバッガビリティを破壊するパターン

型制約を満たせなかったときのエラーメッセージが不可解になる。型パズルとして揶揄される側面の本質。LLM がコードを書ける時代でも、エラーメッセージを読んで判断するのは人間。

### 注意すべきパターン

- **conditional type の分岐深度**: `T extends A ? (T extends B ? ...) : ...` が深いほど、エラー時に「どの分岐で落ちたか」が見えなくなる
- **分配的 conditional type**: `T extends U ? X : Y` で `T` が union だと分配され、エラーメッセージに全分岐の結果が列挙される
- **ユーティリティ型の連鎖**: `Omit<Pick<Partial<T>, K>, J>` のように中間結果に名前がないと、エラーの expected と actual の差分が読めない
- **infer の多段使用**: `T extends (...args: infer A) => infer R ? ...` が入れ子になると、推論失敗時にどの infer が解決できなかったのか特定できない

### 展開コストとの関係

展開コスト（type-expansion-cost）は「型の定義がどれだけ膨らむか」を測る。デバッガビリティは「型の制約を満たせなかったときにエラーメッセージがどれだけ不可解になるか」を測る。相関はあるが同一ではない。

## 2. セマンティクスと構造が乖離するパターン

型の「構造」は正しくても「意味」が消失している。何を表現したいのかが型定義から読み取れない。

### 注意すべきパターン

- **ユーティリティ型の積み重ね**: `Omit<Pick<Partial<T>, K>, J> & Required<...>` のように変換を重ねると、最終的にどんな形のオブジェクトになるか定義から読み取れない。中間結果に名前を付ける（`type` alias や `interface`）ことで意図を明示できる
- **過度に汎用的な型パラメータ**: `T extends Record<string, unknown>` のような広い制約では、型が何を表現するものなのかが伝わらない。ドメイン固有の型で制約する方が意図が明確になる
- **型レベル計算で導出された型**: 計算結果は正しくても、導出過程が定義から追えない場合、使う側は型を「ブラックボックス」として扱うしかなくなる

### ブランド型との対比

`string & Brand<"UserId">` は構造（string）と意味（UserId）が両方読める。これが型設計の良い例。逆に、構造だけが見えて意味が消えている型は危険信号。

## 3. 型チェッカーの計算量を指数的に膨らませるパターン

コンパイル時間が爆発する時限爆弾。要素数が少ないうちは問題にならないため発見が遅れやすい。

### 注意すべきパターン

- **分配的 conditional type + union の組み合わせ**: union のメンバー数に対して分配が発生し、メンバーが増えるたびに計算量が線形以上に増加する
- **再帰的な mapped type**: 型レベルで再帰処理を行うと、再帰深度に応じて指数的に展開される場合がある
- **intersection の多段合成**: `A & B & C & ...` が多いと、型チェッカーがメンバーの互換性チェックで組み合わせ的に計算する

### 展開コストとの関係

展開コストは AST の構造的な複雑さを近似するが、計算量の爆発は「型引数に何が渡されるか」に依存するため、AST だけでは検出が難しい。実行時のコンパイル時間の異常を検知する方がアプローチとして適切かもしれない。
