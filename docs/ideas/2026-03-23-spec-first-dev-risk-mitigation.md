# 仕様書ベース開発のリスクと対策

## 概要

specdrift を活用した仕様書ベース開発（仕様→コード生成→運用）で想定されるリスクと、その対策の整理。「機械で検出 → エージェントが解釈 → 人間が判断」の階層でリスクを段階的に吸収する設計。

## リスクと対策

1. **無言 update** — commit スキルのガード + 対話型 update モード（別アイディア）で構造的に防止
2. **意味的ドリフト（コード変更なしに振る舞いが変わる）** — 原理的な検出限界。テストと人間レビューで対処
3. **仕様→コード生成時の意図のズレ** — バックトランスレーション（コードから仕様を逆生成し元仕様と diff）で検出。behavior の対称的構造（okAs/failAs）が突き合わせ精度を高める。スキルで実装可能
4. **逆方向フロー（バグ修正でコードが先に正になる）** — 不可避なケースとして許容。サブエージェント評価でバイアスを排除し仕様に反映
5. **過剰仕様化** — coverage は「判断のための情報」であり「目標値」ではないという設計思想で対処

## 背景

- specdrift はファイルハッシュによる変更検出を担い、解釈はエージェント側に分離するのがツールの責務境界
- バックトランスレーションは specdrift に組み込まず、エージェントのスキルとして実装するのが適切（ツールのシンプルさを維持）
- このプロジェクトの behavior 宣言が仕様のアンカーポイントになり、仕様書とコードの突き合わせを構造的にやりやすくしている
