# InputScenarios: パスごとの入力シナリオ宣言の設計判断

## 概要

`failAs` / `okAs` の return 文にシナリオラベル（文字列配列）を付与し、`testBehavior` の `parameterize` でそのラベルをパラメータキーとして型レベルで強制する仕組み。実装側がパスごとの入力条件を宣言し、テスト側が具体値を埋める構造になる。

## 設計判断の経緯

### 1. 型制約ではなくテキスト情報にした理由

入力の型構造は全パスで同じ（`BehaviorInput<F>`）。パスごとに異なるのは「name が空」「email が不正形式」といった値の条件であり、型では表現しきれない。テキストラベルで宣言し、キーの一致を型で強制する方式に落ち着いた。

### 2. シナリオの宣言場所を return 文にした理由

各 return 文（`failAs` / `okAs`）が「ここに到達するための入力条件」を最もよく知っている。実装のコントロールフローそのものがシナリオの分岐条件を表しているため、宣言場所として最も自然。`Desc` 型にシナリオ情報を乗せることで、`ExtractByLabel` と同じ仕組みで取り出せる。

### 3. `defineContract` のトップレベルオプションではなく `onInputError` の戻り値にした理由

当初は `defineContract({ inputScenarios: [...] })` として実装したが、シナリオ情報は「パスに付随するもの」であり、パスの所有者（= return 文）がシナリオも所有すべき。`onInputError` を省略した場合（デフォルトの "入力値が不正" パスが自動生成される場合）は、実装者がそのパスを意識していないため、シナリオ宣言も不要。

### 4. `defaultInputError` ヘルパーを用意した理由

`onInputError` 内で `failAs("入力値が不正", "validation_failed", { fields: flatten(issues).nested ?? {} }, [...])` と書く定型部分が冗長だったため、`defaultInputError([...])` で簡潔に書けるヘルパーを contract モジュールに追加した。

### 5. バリデーションエラーで分岐しない理由

Valibot は全フィールドを一括検証して issues 配列で返すため、「name が空 **かつ** email が不正」という複合パターンがありうる。`onInputError` 内で issues を見て分岐すると組み合わせ爆発になる。シナリオラベルは排他性や組み合わせには言及せず、テスト側が具体値を自由に選べる余地を残す。

## 応用候補

- `okAs` / `failAs` のシナリオ宣言は `defineContract` に限らず `defineBehavior` でも使える
- シナリオ宣言がないパスでは `parameterize` のキーが自由（`string` フォールバック）なので、既存コードへの影響なくインクリメンタルに導入可能

## 背景

`parameterize` 関数の導入（同じパスに複数入力パターンをテスト）を起点に、「テスト側が自由にキーを書く」のではなく「実装側がシナリオを宣言してテスト側のキーを制約する」方向へ発展した。`Desc` 型に `InputScenarios` ブランドを交差型として乗せることで、既存の `Desc` / `DescLabel` / `ExtractByLabel` の仕組みを壊さずに拡張できた。
