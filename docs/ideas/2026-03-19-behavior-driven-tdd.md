# behaviorパッケージによるTDDの再定義

## 概要

okAs/failAsラベルが仕様宣言として機能し、testBehaviorがラベル集合からテスト項目を型レベルで導出する。これにより「テストを先に書いて仕様を明確にする」という従来TDDの目的を、実装スタブ + 型システムで達成できる新しい開発フローが可能になる。

## 提案フロー

1. **実装スタブを書く** — okAs/failAsでコードパスのラベルを定義（これが仕様宣言）
2. **テストを書く** — testBehaviorが全ラベルに対応するテストケースを要求（コンパイラが漏れを検出）
3. **実装を埋める** — スタブの中身を実装

## 従来TDDとの違い

- 仕様の所在: テスト名（人間管理） → okAs/failAsラベル（型管理）
- テスト漏れの検出: レビューやカバレッジ（事後） → コンパイルエラー（即座）
- パス追加時: テスト追加を忘れがち → コンパイルが通らない

## 応用候補

- AIコード生成との組み合わせ: スタブ（仕様）を人間が書き、テストと実装をAIが埋めるワークフロー
- チーム開発: スタブをレビューすることで実装前に振る舞いの合意が取れる
- ドキュメント生成: ラベル一覧から振る舞い仕様書を自動生成

## スタブの繋ぎ込みによる層間インターフェース検証

各層のスタブを先に書いて繋ぎ込むと、実装を1行も書かないうちに層間の仕様の過不足が型レベルで見える。

- **下位のパスが多すぎる**: 上位で束ねる判断が明示的になる（例: gateway の `insufficient_funds` と `card_declined` を domain では `charge_failed` に統合）
- **下位のパスが足りない**: 上位のスタブが書けないので気づく（例: domain で「メール送信失敗」を扱いたいが gateway に対応するスタブがない）
- **上位のパスに対応する下位がない**: 実現手段の欠落が見える

各層を単独で設計していると気づきにくい問題が、スタブ同士の接続で自然に浮き上がる。

## Tips: スタブで return を並べる書き方

スタブ段階では実装がないため、複数の return を並べて型だけ定義したい。TypeScript は unreachable な return でも戻り値型の union に含めるので、素朴に並べても型推論は正しく動く。

```typescript
// 動くが unreachable 警告が出る（allowUnreachableCode: false だとエラー）
fn: async (input) => {
  return okAs("決済完了" /* ... */)
  return failAs("在庫切れ", "out_of_stock")
}

// 安全な書き方: declare const TODO で分岐
declare const TODO: boolean
fn: async (input) => {
  if (TODO) return failAs("在庫切れ", "out_of_stock")
  return okAs("決済完了" /* ... */)
}
```

どちらもどうせ消すコードなので、プロジェクトの `allowUnreachableCode` 設定に合わせて好きな方を使えばよい。

## 背景

従来TDDの「テストが仕様を先に定義する」価値を、behaviorの型システムが代替できることに気づいた。テストを先に書かなくても、仕様の明確化とテスト網羅の両方が手に入る。
