# DepsOf による依存コンテキストの合成

## 概要

関数の第一引数（ctx）の型を抽出するユーティリティ型 `DepsOf` を定義し、複数の依存関数が要求するコンテキストを intersection で合成する。

```typescript
type DepsOf<F extends (ctx: any) => any> = Parameters<F>[0]

// ハンドラが必要とする能力の最小集合を型で宣言
const getUserById = (ctx: DepsOf<typeof findUserById> & DepsOf<typeof updateAccessLog>) => ...
```

## 応用候補

- **ctx の細分化時の型制約:** ctx が単一の DbContext ではなく、読み取り専用 / トランザクション必須 / 特定テーブルへのアクセス権限など細分化された場合、`DepsOf` の intersection が「このハンドラに必要な能力の最小集合」を型で表現し、不足があればコンパイルエラーになる
- **依存の可視化:** シグネチャを見るだけで「この関数はどの DB 関数に依存しているか」が型レベルでわかる。`DbContext` と直接書く場合には失われる情報
- **テストとの連携:** env に渡す関数の一覧を DepsOf 的な型から導出できれば、モック漏れの検出に使える可能性がある

## 前提条件

このパターンが機能するには、依存リソースが単一のオブジェクト（ctx）にまとまっている必要がある。現状の設計では依存リソースが増えた場合に引数を追加する方式も選択肢にあり、その場合は `DepsOf` で第一引数だけ抽出しても依存の全体像は取れない。

2つ目の外部リソース（外部 API クライアント等）が必要になった時点で、まず「引数追加か、オブジェクトにまとめるか」を判断し、オブジェクトにまとめる方向を選んだ場合にこのアイデアが活きる。

## 背景

API ハンドラの ctx 型を `Parameters<typeof findUserById>[0]`（間接導出）から `DbContext`（直接記述）に統一するルールを作った際、間接導出には「この関数の依存」という意図表現の価値があることに気づいた。現状は全 DB 関数が同じ DbContext を受け取るため `DepsOf` は実質エイリアスに過ぎないが、ctx が細分化されるユースケースが出てきた時点で再検討する価値がある。
